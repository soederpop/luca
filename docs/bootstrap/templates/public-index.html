What You Build With Luca

Luca is for building traditional software that embeds AI assistants — or supports them. Network services, command-line tools, and executable documents, where an assistant is a first-class part of the program rather than a bolt-on.

Services & APIs

HTTP routes from an endpoints/ folder, WebSocket and IPC socket servers, MCP servers that expose your features as tools to any agent.

CLI Tools

Commands with subcommands and rich generated help, plus everything for a beautiful TUI — chalk colors, inquirer prompts, full React Ink apps.

Notebooks & Runbooks

Markdown files whose code blocks actually run. Docs that execute, demos that can't go stale, runbooks that do the work they describe.

Every kind of app is a thin interface over the same container — and so are your assistants.
The philosophy: the container is a per-process singleton that holds every dependency — features, clients, servers, state, events. Your business logic lives there once. A CLI command, an HTTP endpoint, and an assistant tool are all just different doorways into it. Every project you build accumulates more helpers, and every new project starts richer than the last.

Auto-Discovery the luca cli

You don't register anything. Drop a file in the right folder and the luca CLI finds it before dispatching any command — your commands appear in luca's listing, your features become available via container.feature(), your endpoints become routes.

luca.cli.ts is your customization hook — it runs before any command, so you can seed databases, register extras, or reshape the container for the whole project.
luca                      # list every command — built-ins + yours from commands/
luca describe features    # index of everything the container provides
luca describe fs          # full docs for any helper, generated from source
luca eval "container.features.available"   # poke the runtime directly

Run Anything luca run

luca run executes TypeScript and markdown with the container already in scope. No imports of fs or path, no setup — the container is the standard library.

TypeScript

luca run script.ts

Scripts get container injected. Files via container.feature('fs'), paths via container.paths, utils via container.utils.

Markdown

luca RUNME
luca run docs/guide.md

Fenced code blocks execute top to bottom. Your project ships with RUNME.md — try it now.

Why markdown? A document whose examples run is a document that can't lie. Tutorials become tests, runbooks become automation, and the prose explains the code sitting right next to it. This is the "notebook" in notebooks & runbooks — and it's also how luca prompt works below.

Scaffold luca scaffold

Every helper type has a generator that produces working boilerplate — and a built-in tutorial that teaches the pattern. Scaffolds land in the auto-discovered folders, so the moment the file exists, it's live.

command

A CLI command with a Zod args schema — flags, positionals, and --help generated from it.

luca scaffold command deploy

feature

Reusable business logic on the container — available to commands, endpoints, and assistant tools alike.

luca scaffold feature cache

endpoint

A REST route served by luca serve.

luca scaffold endpoint users

client / server

API clients and long-running servers.

luca scaffold client github

Add --tutorial to any scaffold for the full guide: luca scaffold feature --tutorial

Assistants luca scaffold assistant

An assistant is just a folder. Scaffold one and the pieces are consumed automatically: the prompt, the tools, the lifecycle hooks — no registration, no framework ceremony. 

luca scaffold assistant researcher
The folder is the assistant. CORE.md becomes the system prompt; each exported function in tools.ts becomes a callable tool.

Inside tools.ts, the container is a global — so a tool is three lines away from your whole stack:

// tools.ts — each schema key pairs with an exported function of the same name
export const schemas = {
  searchDocs: z.object({ query: z.string().describe('What to look for') }),
}
export async function searchDocs({ query }) {
  return container.feature('grep').search(query)   // same features as everywhere else
}

Then talk to it:

luca chat researcher      # interactive chat, tools included

Prompts as Programs luca prompt

luca prompt closes the loop: a markdown prompt file can evaluate code first — fenced blocks run against the container, and their output is woven into the prompt — then the result is sent to a coding agent (Claude Code, Codex) or one of your own assistants.

Prompts stop being static text — they gather their own context at send time.
luca prompt claude ./prompts/refactor.md       # send to Claude Code
luca prompt researcher ./question.md            # or to one of your assistants
luca prompt claude a.md b.md --parallel         # run prompts side by side
luca prompt claude ./task.md --dry-run          # preview the resolved prompt first
The payoff: the same markdown format runs as a notebook (luca run) and drives an agent (luca prompt). A runbook that queries your database, embeds the results, and hands the whole thing to an assistant is one file — versioned, reviewable, repeatable.

Content DB docs/ folder

Your docs/ folder is a queryable database of markdown documents. Schemas live in docs/models.ts; frontmatter is validated against them.

// in any script, command, or assistant tool
const docs = container.docs
const posts = await docs.query(docs.models.Post).fetchAll()

Run luca describe contentDb for query syntax and the full API.
